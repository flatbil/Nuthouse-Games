Game 01 — Tap Clicker

Working title: Compound  ·  Engine: Godot 4  ·  Platform: Android (iOS later)

Weekend Build Classroom Demo Solo Dev

Concept

A pure tap/idle incremental game. The player taps a button to generate a resource, buys upgrades to increase passive income, and watches the numbers compound over time. The game is intentionally minimal at launch — the goal is a working pipeline, not a sophisticated game.

Every architectural decision is made to be reused across future titles. This is Game 1 of a catalog, not a one-off project.

The loop is self-reinforcing: upgrades remove the need to tap constantly, which opens time for the player to return later. Offline income is the core retention hook — players return to collect accumulated resources.

Upgrades expand to 8–12 in the first content update. Keep the launch tree hand-authored — no procedural generation until you understand player behavior from analytics.

Project Architecture

game_01/
  project.godot  — engine config, autoload declarations, export settings

  autoloads/  — global singletons, always in memory
    EventBus.gd  — signal hub, decouples all systems
    SaveManager.gd  — sole writer to disk, auto-save timer
    GameManager.gd  — all game state: resources, upgrades, rates

  scenes/
    Game.tscn  — main scene (UI layout)
    Game.gd  — UI logic only, reads from GameManager via EventBus

  assets/  — fonts, icons, audio (mostly empty at launch)

Key Systems — Responsibilities

FileOwnsDoes NOT own
EventBus.gd Signal definitions. Nothing else. Any game logic or state
SaveManager.gd File I/O. Auto-save timer. Load on startup. What data means — just serialize/deserialize
GameManager.gd All game state. Upgrade definitions. Tap logic. Passive rate calculation. Any UI. Any file I/O. Calls SaveManager to save.
Game.gd UI display. User input forwarding. Listens to EventBus signals. Game logic. Never writes state directly — always calls GameManager.

The rule: UI never touches game state directly. Game state never touches UI. EventBus is the bridge. This pattern scales to any size project and makes testing straightforward.

Data Flow

  Player taps button
        │
        ▼
  Game.gd  ──calls──▶  GameManager.tap()
                              │
                        resources += 1
                              │
                        EventBus.resource_changed.emit(amount)
                              │
        ┌─────────────────────┘
        ▼
  Game.gd._on_resource_changed(amount)
        │
        ▼
  ResourceLabel.text = format(amount)
        │
        ▼
  _refresh_upgrade_buttons()   ← checks affordability

GameManager owns the truth. Game.gd only reflects it. If you add a second UI scene later (e.g. a prestige screen), it connects to the same signals without touching GameManager code.
